# Token Stream Visualizer

A lexical analyzer built with **C** and **Flex (Lex)** that reads any C source file and prints a formatted token stream table — complete with token types, values, line numbers, and a summary report.

## What It Does

Given any C source file (or any plain-text file), the visualizer:

- Scans every character using rules defined in `lexer.l`
- Classifies each token (keyword, identifier, operator, etc.)
- Prints a clean ASCII table showing **No. | Token Type | Token Value | Line**
- Prints a **summary table** at the end with counts per category
- Reports unrecognised characters as **ERROR** tokens
- Skips comments (but counts them separately)

## How It Works

The pipeline has three stages:

```
lexer.l  ──(flex)──►  lex.yy.c  ──(gcc)──►  tokenizer  ──(run)──►  output table
```

| Stage | Tool | Input | Output |
|-------|------|-------|--------|
| 1 | `flex` | `lexer.l` | `lex.yy.c` |
| 2 | `gcc` | `lex.yy.c` | `tokenizer` (binary) |
| 3 | `./tokenizer` | `input.txt` | Printed token table |

## Token Categories

| Category | Examples | Description |
|----------|----------|-------------|
| `KEYWORD` | `int`, `if`, `while`, `return` | All 32 standard C keywords |
| `IDENTIFIER` | `main`, `x`, `my_var` | User-defined names |
| `INTEGER` | `42`, `0`, `100` | Whole number literals |
| `FLOAT` | `3.14`, `2.718e+1` | Decimal / scientific notation |
| `STRING` | `"hello"`, `"world"` | Double-quoted string literals |
| `CHAR_LIT` | `'A'`, `'\n'` | Single-quoted character literals |
| `OPERATOR` | `+`, `==`, `&&`, `++`, `->` | All C operators (single & multi-char) |
| `SYMBOL` | `;`, `{`, `(`, `[` | Punctuation / delimiters |
| `ERROR` | `@`, `#`, `` ` `` | Unrecognised characters |
| *(Comments)* | `// ...`, `/* ... */` | Counted but not printed as tokens |

## Build & Run

All commands are run from inside the `Token_Stream_Visulaizer/` directory.

### Option A — Using Make (recommended)

```bash
# Build + run on input.txt in one step
make run

# Or build first, then run separately
make
./tokenizer input.txt
```

### Option B — Manual (step by step)

```bash
# Step 1: Generate C code from the flex rules
flex lexer.l

# Step 2: Compile the generated C file
gcc -Wall -o tokenizer lex.yy.c

# Step 3: Run the tokenizer on your input file
./tokenizer input.txt
```

### Other Make Targets

| Command | What it does |
|---------|--------------|
| `make` | Build the tokenizer binary |
| `make run` | Build and run on `input.txt` |
| `make test FILE=my.c` | Build and run on any file you choose |
| `make clean` | Delete `tokenizer` and `lex.yy.c` |
| `make help` | Show all available targets |

## File-by-File Explanation

### `lexer.l`

This is the heart of the project. A Flex source file has three sections separated by `%%`:

```
[Section 1: Definitions]
%%
[Section 2: Rules]
%%
[Section 3: User C Code]
```

**Section 1 — Definitions**
- `%option noyywrap` tells Flex not to look for a `yywrap()` function (simplifies compilation).
- Named pattern macros like `DIGIT`, `LETTER`, `ID`, `FLOAT_LIT` make rules readable.
- Counter variables (`line_no`, `token_no`, `cnt_keyword`, …) track statistics.

**Section 2 — Rules**
Each rule is a pattern–action pair:
```
PATTERN   { action in C; }
```
Rules are matched in order — longer matches win, and ties go to the rule listed first.
That is why multi-character operators (`==`, `++`) are listed *before* single-character ones (`=`, `+`),
and keywords are listed *before* the identifier catch-all.

**Section 3 — User Code**
- `print_banner()` — decorative header
- `print_header()` / `print_footer()` — table borders
- `print_row()` — one table row per token
- `truncate_val()` — keeps long values from breaking column alignment
- `print_summary()` — final statistics table
- `main()` — opens the file, sets `yyin`, calls `yylex()`, then prints the summary

### `input.txt`

A self-contained C program that deliberately exercises every token category:
- All common keywords (`int`, `float`, `char`, `struct`, `typedef`, …)
- Integer and float literals (including scientific notation)
- String and character literals
- All operator families (arithmetic, relational, logical, bitwise, compound-assignment)
- Single-line (`//`) and multi-line (`/* */`) comments
- Intentional error characters (`@`, `#`, `` ` ``) at the end

### `Makefile`

Automates the two-step build process and provides convenience targets.
The key reason we don't need `-lfl` (the Flex library) is that `%option noyywrap`
is declared in `lexer.l`, removing the only dependency on that library.

## How Flex Works (Theory)

Flex converts your pattern rules into a **Deterministic Finite Automaton (DFA)**:

```
Source characters
      │
      ▼
  ┌────────────┐
  │  DFA built │  ◄── generated from your regex rules in lexer.l
  │  by Flex   │
  └─────┬──────┘
        │  longest match wins
        ▼
   Token identified
        │
        ▼
   Action executed (printf, counter update, etc.)
        │
        ▼
   Next character scanned ...
```

Key Flex concepts used in this project:

| Concept | What it means |
|---------|---------------|
| `yytext` | The matched text (a `char *`) |
| `yyleng` | Length of the matched text |
| `yyin` | Input file pointer (defaults to `stdin`) |
| `yylex()` | The main scanning function — call it once to process the whole file |
| `%option noyywrap` | Tells Flex the input is a single file; no chaining needed |

## Common Errors & Fixes

| Error | Cause | Fix |
|-------|-------|-----|
| `flex: command not found` | Flex not installed | `brew install flex` or `sudo apt install flex` |
| `gcc: command not found` | GCC not installed | Install Xcode CLI tools or `sudo apt install gcc` |
| `cannot open file 'input.txt'` | Wrong working directory | `cd Token_Stream_Visulaizer` first |
| `lex.yy.c: No such file` | Flex step was skipped | Run `flex lexer.l` before `gcc` |
| Unmatched rule warnings | Rule order issue | Make sure multi-char operators come before single-char ones |
| `-lfl` linker error | Missing flex library | Use `%option noyywrap` in `lexer.l` (already set) and drop `-lfl` |
